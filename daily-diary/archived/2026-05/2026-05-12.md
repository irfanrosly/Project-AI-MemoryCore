# Daily Diary — 2026-05-12
*MYQuitMate dev session — admin access debugging*

## Session Summary
**Date**: 2026-05-12  
**Session Type**: Bug investigation + production fix  
**Project**: MYQuitMate (my-quit-fe / my-quit-be)

## Main Topics

1. **Admin redirect not working on prod**: After login, admin user not redirected to `/admin`. Investigated FE and BE.

2. **Root cause found — backend code not deployed**: `auth.service.ts` had `role: user.role` added to login/register/refresh responses locally but NEVER committed. Railway was running old code that omitted `role` from the response. Frontend login page checks `user.role === 'ADMIN'` — with `role` missing, fell through to `/dashboard`.

3. **Other BE fixes committed**: Also included were `role` in JWT strategy validate return, `role` in `/users/me` select fields, and gamification badge check on `GET /badges`.

4. **Localhost fix**: `.env.local` was missing `BACKEND_URL=http://localhost:3001` — BFF proxy at `app/api/auth/[...path]/route.ts` uses this env var. Without it, all auth calls on localhost fetched `undefined/auth/...` and crashed.

## What Got Done
- Added `BACKEND_URL=http://localhost:3001` to `.env.local` — localhost auth now works
- Committed all uncommitted BE changes (role in responses, jwt strategy, users service, gamification)
- Merged `feature/smoke-logging` → `main` on backend, pushed → Railway redeploy triggered
- Prod should now return `role: "ADMIN"` on login → redirects to `/admin` correctly

## Key Technical Insight
Subtle bug pattern: local dev and production using SAME DB (Supabase), but backend code diff between local and Railway caused divergence. DB role was correct; response serialization was wrong. Always commit and deploy backend changes before debugging FE redirect logic.
